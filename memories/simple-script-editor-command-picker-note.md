**Intermittent picker→editor drag failure was a host-app CSS bug, not anything in this repo.** Symptom: dragging from the picker into the editor worked on first open, but after doing an in-editor reorder drag (or sometimes even after just refocusing/reopening the picker) it silently stopped working — `[picker] dragStart` still fired, but *nothing* on the editor's side did, not even our own delegated `#listContent` `dragover`/`drop` listeners. Root-caused via the same console.log-at-every-step technique used for the earlier close-button/drag bugs: a temporary `document`-level capture-phase `dragenter` logger showed the failing drag's pointer passing over a `<div class="resize-handle resize-handle-w" style="left:0px; top:12px; height:401px;">` on its way from the picker's window into the editor's — a resize-grip belonging to the host app's window/docking system (`dock-spawn-ts`, found via the user's host-app path `D:\kcc_dev\HTML`, same repo as the earlier `openModal` fix — see `MccVisualizationHandler.ts`). Traced to two combined issues, both fixed in the **host app**, not `web-component-designer`: (1) `dock-spawn-ts`'s `ResizeHandle.adjustSize()` (`node_modules/dock-spawn-ts/src/ResizeHandle.ts`) only ever sets *one* axis via inline style per handle (`height` for west/east handles, `width` for north/south) — the perpendicular "thickness" axis is expected to come from a CSS base class, by design; (2) the host's own `assets/css/mlog/dock/dockspawn.css` never defined that base class — it had only an empty `.resize-handle-corner {}` and no `.resize-handle` rule at all, so the unset axis had literally no defined size, letting a west-edge resize handle render far wider than its intended ~6px and intercept drag gestures meant for content near a window's edge. Fixed by adding `.resize-handle { width: 6px; height: 6px; }` to that stylesheet (matches `ResizeHandle.ts`'s own `handleSize = 6` constant). **Lesson**: when a drag/drop bug involves two separate floating dialog windows and reproduces inconsistently depending on window focus/position, suspect the host's window-manager/docking chrome intercepting the native drag gesture geometrically — log at the `document` level in capture phase to see every element the drag passes over, not just the ones inside your own component.

**Drag-and-drop directly from `SimpleScriptCommandPicker` into `SimpleScriptEditor`'s list, at the exact drop position**: the picker's tree (`SimpleScriptCommandPicker.ts` constructor) has a minimal `dnd: { guessDropEffect: false, serializeClipboardData: true, dragStart: (e) => { e.event.dataTransfer.effectAllowed = 'copy'; e.event.dataTransfer.dropEffect = 'copy'; return true; } }` — no drag-handle restriction like the editor has, since picker rows have no competing buttons; `guessDropEffect` is a *required* field on `DndOptionsType` even though only `dragStart`/`serializeClipboardData` are otherwise needed (this tree is drag-source-only, never a drop target — no `dragEnter`/`dragOver`/`drop` needed on the picker's own tree). Cross-tree drops are allowed by Wunderbaum by default (`preventForeignNodes` defaults to `false`; native drag/drop events are `composed: true` and cross shadow-DOM boundaries fine — dragging between two separate custom elements/shadow roots in the same document just works at the browser level).

**Gotcha: `e.sourceNode` is `null` for any cross-tree drop, not just "the foreign one" — crashed on first real-world test** (`TypeError: Cannot read properties of null (reading 'tree')` inside our own `dragEnter`). Each Wunderbaum instance's `DndExtension` tracks `this.srcNode` purely from *its own* "dragstart" handling; the *target* tree's extension instance (e.g. the editor's, receiving a drop that started in the picker's completely separate Wunderbaum instance) never saw that dragstart and so has no way to know the source node object at all — `e.sourceNode` is unconditionally `null` there, it does **not** carry a reference to the foreign node with `.tree` pointing elsewhere as first assumed. This means `e.sourceNode` itself *is* the same-tree/foreign-tree signal (truthy = reordering within this tree, `null` = dragged in from elsewhere) — checking `e.sourceNode.tree === this._commandListFancyTree` was wrong twice over (both the null-deref crash, and the premise that a foreign node reference would even be present to compare). Fixed by branching on `if (e.sourceNode) { ...moveTo()... } else { ...use e.sourceNodeData... }` in the editor's `dragEnter`/`dragOver`/`drop`. For the `else` branch, the dragged command's type name comes from `e.sourceNodeData?.title` — Wunderbaum's own drop handling (`wb_ext_dnd.ts`) separately serializes the dragged node via `e.dataTransfer.setData(nodeMimeType, JSON.stringify(srcNode.toDict(...)))` on dragstart (only if the *source* tree's `dnd.serializeClipboardData` is truthy — hence that flag on the picker) and deserializes it into `sourceNodeData` (a plain `WbNodeData`-shaped object, `{title, ...}`) on drop, specifically to support exactly this kind of cross-instance drag. `drop`'s foreign branch builds a brand-new command (`{ type: e.sourceNodeData.title }`) via `createTreeItem()` + `addChildren()` then `moveTo(e.node, region)` to land it at the precise drop position — deliberately never calls `moveTo()` on anything from the picker's tree itself (impossible anyway since we have no reference to it, and semantically wrong regardless — we want a *copy*/new instance, not to move a catalog entry).

**Dropping in the empty area below the last row (not onto any row) is handled separately, outside Wunderbaum's own dnd system entirely.** Wunderbaum only resolves a drop target when the event lands on an actual `.wb-row`; if `Wunderbaum.getNode(e)` finds nothing (empty space below the last row), its internal `onDropEvent` returns immediately at the top (`if (!targetNode) { this._leaveNode(); return; }`) *without* calling `stopPropagation()` — unlike the row-targeted "drop" branch, which does call `stopPropagation()`. That asymmetry is exactly what makes a workaround possible: our own plain native `dragover`/`drop` listeners, delegated on the stable `#listContent` wrapper (same reason as the mousedown-handle listener — `#commandList` itself gets replaced by `destroy()`), only ever receive events that *weren't already row-handled and stopped*, so there's no need to manually check "is this a row" ourselves — by construction, anything that reaches our delegated listener is the empty-space case. `dragover` there just calls `preventDefault()` unconditionally (native DnD requires this for `drop` to fire at all on a target with no default draggable behavior) and sets `dropEffect`; `drop` does the actual work, appending to the end of `this._commandListFancyTree.root`. For *which* node/type to append, same-tree vs. foreign is resolved differently than the row-targeted case, since there's no Wunderbaum-provided `e.sourceNode`/`e.sourceNodeData` in a plain native event: a new field `this._draggedNode` is set in the editor's own `dnd.dragStart` (`e.node`, the node about to be dragged — only ever true for a drag that started on this tree) and cleared in a new `dnd.dragEnd` callback (so it can't leak stale into an unrelated later foreign drag); if set at drop time, `this._draggedNode.moveTo(lastChild, 'after')`. If *not* set (foreign drag, e.g. from the picker), the dragged type name is read via the standard-MIME-type fallback `ev.dataTransfer.getData('text/plain')` — Wunderbaum *always* sets this to the dragged node's `.title` during its own dragstart handling (`if (!e.dataTransfer.types.includes('text/plain')) e.dataTransfer.setData('text/plain', srcNode.title)`, unconditional, regardless of `serializeClipboardData`), so no extra wiring was needed on the picker's side for this path specifically. Deliberately *not* reused for the row-targeted same-tree case (which needs the actual node reference, not just a title string, since duplicate titles are common — e.g. two "Delay" rows in the same list — and title-matching would be ambiguous there).

Known limitation (not addressed): dropping still requires the target tree to have at least one existing row to compute a drop *region* against for the row-targeted (before/after) case — but the empty-space fix above means a fully empty list can now still receive a drop (append path triggers directly, no row needed), so this limitation is effectively gone for the common case; "Add Command" remains the only path if you want to *not* drag at all.

Added a small italic hint (`<span class="hint">or drag a command into the script list</span>`) next to the picker's "Add" button, so the drag-and-drop path is discoverable, not just the button.

**`openModal`'s `abortSignal` doesn't actually close the window in this host app — worked around, not fixed, since the bug lives outside this repo.** Diagnosed by temporarily logging every step of `addCommand()`'s open/close flow when it used `openModal`. Console output across a full open→click-Close session, in order: `"opening modal"` → `"openModal resolved (dialog closed)"` (!) → *(much later, on the actual Close click)* → `"[picker] closeClicked fired"` → `"[editor] closeRequested received, aborting"` → `"[editor] abortSignal fired"`. Two things this proved: (1) `openModal`'s returned promise resolves **immediately upon opening**, not upon closing — unlike `openConfirmation`, `await`ing it does not block until the dialog is dismissed. (2) Our entire event chain (button → `closeRequested` → `abortController.abort()` → `'abort'` fires) was 100% verified correct and complete on our side — yet the window stayed open, meaning the host app's `openModal` implementation accepts `abortSignal` (matching the interface) but doesn't actually subscribe to it to close the window, unlike its `openConfirmation` implementation which demonstrably does. **Fix that actually works**: `addCommand()` was switched back to `openConfirmation` (`confirmText`/`cancelText` both `"Close"`, since Add/Close are handled entirely by the picker's own toolbar and these host buttons are now redundant fallbacks) purely because its `abortSignal` handling is proven reliable — `picker.title = "Add Script Command"` also had to come back since `openConfirmation` has no `title` option and reads `.title` instead. If the host app's `openModal` is ever fixed to honor `abortSignal`, switching back would be straightforward — but don't assume it works without re-verifying via the same logging technique.

**Gotcha: `@click="[[...]]"` template bindings silently do nothing without a `ready()` hook**: right after adding the Add/Close buttons, clicking them did nothing (no error) while Wunderbaum's own `dblclick` callback worked fine. Root cause: `SimpleScriptCommandPicker` only had a `constructor()`, no `ready()` — and `_assignEvents()` (the framework call that actually wires up `@click="[[this.method()]]"` template bindings, see `BaseCustomWebComponentConstructorAppend`) is **not** invoked automatically; every other component in this package that uses `@click` bindings (`SimpleScriptEditor`, etc.) calls `this._parseAttributesToProperties(); this._bindingsParse(null, true); this._assignEvents();` from its own `ready()`. This had gone unnoticed because the picker previously had zero template event bindings (only inline `.onkeyup`/Wunderbaum JS callbacks, which don't need this). **Lesson**: any new custom element in this codebase that adds its first `@click`/similar template binding must also add a `ready()` calling these three methods, or the binding is silently inert.

**Drag handle hit-area regression (root-caused via runtime diagnostics, not static reading)**: after several rounds of unrelated edits, dragging silently stopped working — cursor showed a valid "move" indicator but nothing ever reordered, with no console error. Extensive static review of the `dnd` config and Wunderbaum's internal `moveTo()`/row-positioning code (rows are `position:absolute` with `style.top` set per index, not DOM-reordered — `_updateRows()` in `wunderbaum.ts`) found nothing wrong there. Root cause only surfaced by temporarily logging `dragStart`'s rejected `event.target`: it was `span.wb-node` itself (the whole row), not `.drag-handle`.

**The real root cause was structural, not cosmetic**: `git show HEAD:.../SimpleScriptEditor.ts` (the pre-session baseline) had `dragStart` unconditionally `return true` — no handle restriction existed before this feature was added, and it worked. The handle-restriction check (`e.event.target.closest('.drag-handle')`) was broken *by design* from the moment it was introduced: per the HTML5 DnD spec, `dragstart`'s `event.target` is defined to be the **draggable element itself** (`nodeElem`, since that's what has `draggable=true`), never the specific descendant that was actually pressed — `.closest()` searches upward from that target, so it can never reach a *descendant* class like `.drag-handle` no matter how big or well-positioned the handle is. Enlarging the handle to a full-row-height strip (see below) never could have fixed this on its own, and a first attempt to fix the sizing (appending the handle to `rowElem` as a sibling of `nodeElem` instead of a descendant) made things actively worse, since HTML5 drag only initiates from the draggable element **or its descendants** — a sibling isn't a descendant, so pressing it never triggered `dragstart` at all, falling back to native browser selection-drag (visible as a blue highlight).

**Correct, working fix** (two independent parts, both needed):
1. `.drag-handle` is a `position:absolute; left:0; top:0; height:100%; width:24px` full-row-height strip (like `.cmd` on the right) for a reasonably-sized click target, but inserted via `e.nodeElem.prepend(leadingSpacer, handle)` — i.e. kept as a **child of `nodeElem`**, not a sibling in `rowElem`. A matching in-flow `leadingSpacer` (24px) reserves its width so title text doesn't render underneath it.
2. Since `dragstart`'s `event.target` can never be used to detect which descendant was pressed, detection was moved to `mousedown` (whose `event.target` *does* correctly reflect the actual innermost clicked element) — a `mousedown` listener on the stable `#listContent` wrapper (chosen because `#commandList` itself gets replaced by Wunderbaum's `destroy()`) sets `this._dragFromHandle = !!target.closest('.drag-handle')`; `dnd.dragStart` then checks `this._dragFromHandle` instead of inspecting its own event's target at all.

**Follow-up: dragged row stuck at 50% opacity after drop**: Wunderbaum adds a `wb-drag-source` class (`.wb-row.wb-drag-source { opacity: 0.5; }` in `wunderbaum.css`) on drag start and normally removes it on the native `dragend` event (`wb_ext_dnd.ts`). In practice this removal can race against our own `drop` handler's `moveTo()` (which itself schedules a re-render via its own `setTimeout(0)`, per a comment in `wb_node.ts` acknowledging dragend timing is delicate) and lose, leaving the moved row visually stuck grayed-out. Fixed defensively by calling `e.sourceNode.setClass('wb-drag-source', false)` explicitly right after `moveTo()` in our own `drop` handler, instead of relying on native `dragend` cleanup timing.

First attempted fix was wrong and made things *worse* (dragging degraded into an inconsistent native text/element-selection drag, blue-highlighting the row instead of reordering): made `.drag-handle` a `position:absolute; left:0; top:0; height:100%; width:24px` full-row-height strip like `.cmd`, but appended it to `rowElem` (`e.nodeElem.parentElement`) as a **sibling** of `nodeElem` — mirroring `.cmd`'s placement. That's wrong specifically for the drag handle: Wunderbaum sets `nodeElem.draggable = true` (`wb_node.ts`) on `nodeElem` itself, and HTML5 native drag-and-drop only initiates from a draggable element **or its descendants** — a sibling, even one visually overlapping and stacked on top of `nodeElem`, is not a descendant, so pressing down on it never triggers `dragstart` at all; the browser falls back to default mouse behavior (selection-drag). It "sometimes worked" only when the overlap/stacking happened to let a click fall through to `nodeElem` underneath by coincidence — inherently unreliable, not a real fix.

Correct fix: keep `.drag-handle` absolutely positioned/full-row-height (still resolves its containing block up to `.wb-row`, since neither it nor `nodeElem` needs its own `position` for that), but insert it as a **child of `nodeElem`** (`e.nodeElem.prepend(leadingSpacer, handle)`), preserving descendant-of-the-draggable-element status. A matching in-flow `leadingSpacer` (24px) is prepended alongside it so the title text doesn't render underneath the absolutely-positioned handle; the reserved-row-width calc sums `leadingSpacer.offsetWidth + spacer.offsetWidth`. **Lesson (two-fold)**: (1) when a DnD/click regression shows no exception and the internal library logic looks unchanged, add throwaway `console.log`s at the entry/rejection points (e.g. `dragStart`'s early-return checking `event.target`) before spending more time re-reading library internals. (2) `.cmd`'s "append to `rowElem` as a sibling of `nodeElem`" pattern is fine for plain clickable buttons, but is NOT a safe template to copy for anything that needs to inherit `draggable` — drag sources must stay nested inside the actual `draggable=true` element.

Reordering was first tried as explicit Move-up/Move-down buttons (reusing `assetsPath + 'icons/moveLeft.svg'` rotated 90/270deg like `ZMoveContextMenu.ts`'s z-order buttons), but the user found that less intuitive than a dedicated drag handle and asked to drop the buttons in favor of one. Current approach: a small "3 horizontal lines" inline-SVG handle (`.drag-handle` span) is prepended at the very front of each row via `e.nodeElem.insertAdjacentElement('afterbegin', handle)`. Wunderbaum sets `draggable=true` on the whole row (`wb_node.ts` `nodeElem.draggable = true`), so dragging is restricted to the handle by checking `e.event.target.closest('.drag-handle')` inside the `dnd.dragStart` callback and returning `false` (which calls `e.preventDefault()` internally, per `wb_ext_dnd.ts`'s `onDragEvent`) when the drag didn't start on the handle — the row itself stays draggable-attributed, only the *initiation point* is gated.

**Row-width gotcha (two separate causes, both needed fixing)**:
1. Wunderbaum hard-sets `titleSpan.style.width` on every render pass to exactly fill the row/column width (`wb_node.ts` around line 1820, `columns[0]._widthPx - _ofsTitlePx - TITLE_SPAN_PAD_Y`), with no awareness of extra elements a custom `render` callback inserts into `nodeElem` (`span.wb-node`). Prepending the drag-handle and appending the button-strip spacer overflows the row by their combined width. Fix: on `e.isNew`, stash `handle.offsetWidth + spacer.offsetWidth` on the row element (`e.nodeElem._reservedRowWidth`); then on *every* render call (not just `isNew`, since Wunderbaum recomputes `titleSpan.style.width` on each resize/redraw), subtract that reserved amount from `titleSpan.style.width` after Wunderbaum sets it.
2. Separately, Wunderbaum calls `this.element.classList.add("wunderbaum")` directly on whatever element is passed as `element:` in the options (`wunderbaum.ts` line 305) — it does **not** wrap it in a new child div. So `#commandList` itself becomes `div.wunderbaum`, which per `wunderbaum.css` has `border: 2px solid var(--wb-border-color)`. The stylesheet's own `div.wunderbaum * { box-sizing: border-box; }` rule only targets *descendants* (the `*` combinator), not `.wunderbaum` itself, so `#commandList` keeps the browser-default `content-box` sizing. With our own inline `width:100%`, content-box adds the 4px border *on top* of that 100%, making `#commandList`'s outer box wider than its containing pane and triggering a horizontal scrollbar. Fix: `#commandList { box-sizing: border-box; }` in `SimpleScriptEditor.style` — targets only our own element by ID (specificity beats `div.wunderbaum`), doesn't touch the border's appearance or any of wunderbaum's already-border-box descendant styling.

Both fixes were required together; fixing only one left a residual scrollbar.

**Code view toggle**: `SimpleScriptEditor` reuses `CodeViewMonaco` from `@node-projects/web-component-designer-codeview-monaco` (already a direct dependency of this package, pulled in transitively by `BindingsEditor.ts`'s single-line JS expression editors — no new dependency needed) for a "Code"/"Visual" toggle button in the toolbar. `CodeViewMonaco` only creates its actual Monaco instance lazily via a `ResizeObserver` once its container has non-zero width (see `code-view-monaco.ts` `ready()`), so toggling the `.visible`/`.hidden` CSS classes (not raw inline styles — see layout note below) is what triggers first-time creation; `.activated()` (calls `monacoEditor.layout()`) is called via `requestAnimationFrame` after every subsequent show to force a relayout, since Monaco's own `automaticLayout` polling can be slow to notice a hidden→visible flip on an already-created instance.

**Toolbar layout — final settled structure (went through several iterations, see history below)**: the toolbar (Add Command / Code+Visual toggle / Help) lives *inside* `.list-pane` (the split-view's left column), as a sibling *below* a `.list-content` wrapper — `.list-pane { display:flex; flex-direction:column }` with `.list-content { flex:1 1 auto }` (holding `#commandList` and the code-view-monaco, both `position:absolute; inset:0`, toggled via `.hidden`/`.visible` classes) and `.toolbar { flex:0 0 auto; width:100% }` below it. This makes the toolbar naturally match the list column's width (no JS-driven width sync needed) and its bottom edge naturally aligns with the grid-pane's bottom edge (both panes are `height:100%` of `.content`), per user request. Switching to code view only toggles `#commandList.hidden` and `.code-view.visible` *inside* `.list-content` — it does **not** hide `#splitView`/`.list-pane`/`.toolbar` at all anymore, so the toggle button is structurally never at risk of disappearing.
  - *Rejected intermediate approach*: an earlier iteration hid the *entire* `#splitView` (`display:none`) when switching to code view, which also hid the toolbar nested inside it — no way to switch back. Fixed by pulling the toolbar out as a full-width sibling of `.content`, but then it visually spanned the whole width instead of just the list column, and a `ResizeObserver`-driven width sync (`toolbar.style.width = listPane.getBoundingClientRect().width + 'px'`) was added to fake the alignment — fragile (had to special-case width `0` while hidden, since `#listPane` reports 0 width when its hidden ancestor collapses it, which briefly collapsed the toolbar to nothing). The current structure (toolbar nested in list-pane, only `#commandList`/`.code-view` toggle inside `.list-content`) avoids all of this by construction — mentioned here so this dead end isn't retried.

**`Wunderbaum.destroy()` replaces its element via `outerHTML` self-assignment — cached element references go stale**: `destroy()` (`wunderbaum.ts` ~line 1065) does `this.element.outerHTML = this.element.outerHTML;` as its way of stripping all event listeners — this is the classic "clone via outerHTML" trick, and it silently **replaces the DOM node with a new clone** at the same position, detaching the original node from the document. `SimpleScriptEditor._commandListDiv` was cached once in the constructor via `_getDomElement('commandList')`; after calling `this._commandListFancyTree.destroy()` (added when the code-view toggle made `loadScript()` callable more than once per instance), that cached reference pointed at the now-detached old node, so the *new* Wunderbaum instance in the next `loadScript()` call rendered into an invisible, disconnected element — while the real, visible `#commandList` in the document (the new clone) stayed empty forever. This is exactly why commands "disappeared" switching back from code view. Fix: in `loadScript()`, after calling `destroy()`, re-fetch `this._commandListDiv = this._getDomElement('commandList')` before creating the replacement `Wunderbaum` — `_getDomElement()` (`BaseCustomWebComponent.js`) does a live `shadowRoot.getElementById(id)` lookup each call, not a cached one, so this reliably gets the current live node (its `id` survives the outerHTML clone). General lesson: any cached reference to an element that a `Wunderbaum` instance's `destroy()` was called on must be re-fetched afterward, not reused.

**Applying code-view edits**: `_parseCodeViewScript()` (JSON.parse + `Array.isArray(parsed.commands)` check, `alert()` + `return null` on failure) is a pure parse/validate helper with no side effects, used by two call sites that each control `loadScript()`/visibility ordering themselves: `toggleCodeView()`'s switch-back branch calls `_showCodeView(false)` (reveal) *then* `loadScript(parsed)` — reveal-before-rebuild matters so Wunderbaum computes widths against a properly sized container (see below) — and `getScriptCommands()` calls `loadScript(parsed)` directly without touching visibility when `_showingCode` is true, since the dialog's outer "OK" button (`EventAssignment.ts`) calls `getScriptCommands()` directly without going through the toggle button and the dialog is closing anyway (visual correctness doesn't matter, only the returned data does). Without this, JSON typed in the code view but never explicitly switched back out of would be silently discarded on OK.

**Reveal-before-rebuild gotcha**: switching back from code view must call `_showCodeView(false)` (which un-hides `#commandList`) *before* `loadScript(parsed)` rebuilds the Wunderbaum tree, not after — Wunderbaum's column-width math (`vpWidth = this.element.clientWidth`, used to size `titleSpan.style.width`) runs at tree-construction time, and if `#commandList` is still `display:none` then, it computes everything against a zero-size container, leaving rows invisible even after the container becomes visible afterward (nothing re-triggers Wunderbaum's layout on visibility change, unlike Monaco's `automaticLayout`).

This is also the first time `loadScript()` can run more than once on the same `SimpleScriptEditor` instance (previously each instance was only ever loaded once by `EventAssignment.ts`). Wunderbaum's constructor does `this.element.innerHTML = ...` (safe to re-target the same div), but the old instance's own internal listeners aren't cleaned up unless `.destroy()` is called first — so `loadScript()` now starts with `this._commandListFancyTree?.destroy();` before creating the replacement tree.

**Dead expander/indent space per row**: Wunderbaum always reserves a fixed-size (`--wb-icon-outer-width/height`, 20×20 by default) `<i class="wb-expander wb-indent">` slot at the start of every row (`wunderbaum.css` ~line 239), even for a completely flat tree with no nested children — it's just empty space with nothing to expand. Since the command list here is intentionally always flat, this was pure dead space pushing the drag-handle away from the row's left edge. Hidden with `#commandList i.wb-expander, #commandList i.wb-indent { display: none; }` in `SimpleScriptEditor.style` — the `#commandList` ID selector outranks `wunderbaum.css`'s own `div.wunderbaum span.wb-node i.wb-expander` rule on specificity (ID beats type+class combos) without needing `!important`.

**Found the actual root cause of the intermittent picker→editor drag failure (not the resize-handle CSS)**: it was a stale-state bug entirely inside `SimpleScriptEditor.ts`/Wunderbaum, unrelated to the host app. Wunderbaum's `DndExtension.onDragEvent` (`node_modules/wunderbaum/src/wb_ext_dnd.ts`) resolves the dragged node from the DOM element under `e.target` on **every** drag-lifecycle event including `dragend`, and bails out early (`if (!srcNode) { this.tree.logWarn(...); return; }`) *without* clearing its internal `this.srcNode` field if that lookup fails. Since Wunderbaum's rows are position-recycled absolutely-positioned divs (not re-created), a same-tree reorder's own `drop` handling (`moveTo()`) triggers a re-render that can rebind the very row element the drag started on to a *different* node's data before the browser's own trailing `dragend` event fires and gets resolved back to a node — so the lookup fails, the early-return skips resetting `this.srcNode`, and Wunderbaum's internal "what's currently being dragged" state stays stuck pointing at the old node indefinitely. Every *later* `dragenter`/`dragover`/`drop` — including a completely unrelated foreign drag started fresh from the picker — then receives that stale non-null `e.sourceNode` from Wunderbaum, so our own code (which branched on `e.sourceNode ? "move" : "copy"`) incorrectly treated a picker→editor drag as a same-tree move. Setting `dropEffect = "move"` while the *actual* drag's `effectAllowed` was still `"copy"` (fixed at the picker's `dragstart`, per the HTML5 DnD spec `effectAllowed` can never change after `dragstart`) is exactly what the browser renders as a "no-drop" cursor — confirmed by a diagnostic log showing our own `dragover` handler firing correctly and calling `preventDefault()` every time, yet the cursor stayed blocked regardless, since `preventDefault()` alone doesn't matter if the requested `dropEffect` isn't inside the drag's `effectAllowed` set. Matches the user's exact repro steps (in-editor reorder drag, then switch back to the picker, then picker→editor drag stops working) perfectly.
  - **Fix**: stopped trusting Wunderbaum's own `e.sourceNode` (in the tree's row-level `dnd.dragEnter`/`dragOver`/`drop` callbacks) entirely, since it's subject to this staleness bug. Replaced every usage with `this._draggedNode` — a field this component already tracked itself, set from `e.node` in the tree's own `dnd.dragStart` — since that reference comes from the *live* `WunderbaumNode` object (stable across row-recycling; only the bound DOM elements get reused, not the node instances) rather than from a DOM-element-lookup performed late, at `dragend` time. For the foreign-drop case, read the dragged type directly via `e.event.dataTransfer.getData("text/plain")` instead of Wunderbaum's `e.sourceNodeData` (unaffected either way, just consistent with the `#listContent`-level empty-area handler which already worked this way).
  - Made `this._draggedNode`'s own clearing robust too, for the identical underlying reason: the tree's `dnd.dragEnd` callback (which used to be the only place clearing it) is *also* gated behind Wunderbaum's `onDragEvent` successfully resolving a node from the `dragend` event target, so it's vulnerable to the exact same silent-failure-to-fire as Wunderbaum's own internal reset. Added a `document`-level, capture-phase, native `"dragend"` listener (added once in the constructor, alongside the `mousedown` listener that sets `_dragFromHandle`) that unconditionally does `this._draggedNode = null` — this bypasses Wunderbaum's node-resolution entirely, since it doesn't care *which* node the drag was, only that some OS-level drag gesture has now ended.
  - **Lesson**: when a Wunderbaum-provided piece of drag state (`e.sourceNode`, `e.sourceNodeData`) is used across two *different* Wunderbaum instances and the *symptom only appears after an in-tree reorder happened first*, suspect its own internal bookkeeping going stale from a `dragend`-time node-resolution failure caused by row/DOM recycling — don't assume `e.sourceNode` is a fresh, per-drag-gesture value just because it's passed as a callback argument each time.

**Follow-up: the `_draggedNode` fix above was still not fully reliable** — user report: "beim 1x drag versuch nachdem ich ein element verschoben habe geht es noch nicht, aber bei den weiteren funktioniert es" (the very first drag attempt right after moving/reordering an element still fails, but subsequent attempts work). This means `this._draggedNode` itself, while more reliable than Wunderbaum's `e.sourceNode`, is *still* an instance field that only gets reliably cleared *eventually* (via the document-level `dragend` listener) — there's an inherent race for the very next drag attempt if it starts before that clearing has actually run (e.g. `dragend` for the previous gesture firing slightly after the browser starts processing the next gesture's `dragenter`/`dragover`). Any plain-field-based approach ("set at dragstart, cleared at dragend") is fundamentally racy this way, no matter how many redundant clear-listeners are added.
  - **More robust fix**: stopped relying on *any* instance field to decide same-tree-move vs. foreign-copy at `dragenter`/`dragover` time. Instead, the editor's own tree's `dnd.dragStart` now also calls `e.event.dataTransfer.setData(SAME_TREE_MOVE_MIME, "1")` (a custom marker MIME type, `"application/x-simple-script-move"`, module-level constant at the top of `SimpleScriptEditor.ts`) alongside setting `_draggedNode`. Every `dragEnter`/`dragOver`/`drop` (both the row-level `dnd` config and the `#listContent`-level empty-area handlers) now checks `e.event.dataTransfer.types.includes(SAME_TREE_MOVE_MIME)` as the *sole* source of truth for "is this specific gesture a same-tree reorder" — `dataTransfer.types` is a live, browser-owned property of the *current* native drag operation (its *type names* are readable throughout `dragenter`/`dragover`, only `getData()`'s actual *values* are restricted until `drop`), so it is tied 1:1 to the actual in-progress gesture and cannot carry state over from an earlier, already-finished drag the way any field on `this` can, no matter how many places try to clear it. `this._draggedNode` is now only *read* once `isMove` is already confirmed true by the marker check — purely to get the actual node reference for `.moveTo()`, never as the decision signal itself. The document-level `dragend` listener that clears `_draggedNode` was kept (harmless cleanup), but the correctness of the cursor/drop-effect logic no longer depends on it.
  - **Lesson**: for "which of two mutually-exclusive things is this drag" decisions that must be evaluated on every `dragenter`/`dragover` of a live gesture, prefer encoding the answer directly into that gesture's own `dataTransfer` (via a custom MIME type set at its `dragstart`) over any `dragstart`-set/`dragend`-cleared instance field — the former is inherently scoped to exactly one gesture with no possibility of a clear-timing race between two back-to-back drags.

**Hiding the grid-pane (property grid) in code mode**: `@node-projects/splitview.webcomponent`'s `<node-projects-split-view>` host is `:host { display: flex; }` (`SplitView.js`), and it manages pane sizing by writing an *inline* `element.style.flex = '1 1 ' + flexBasis + 'px'` on its two direct children (`_setFlexBasis`) — so a plain CSS `width: 48%/52%` on `.list-pane`/`.grid-pane` is really only the pre-JS-managed starting point; live sizing is inline-flex-basis-driven. This means: (a) you cannot just fight the split ratio with a CSS width override without `!important`, but (b) you *don't need to* — with `flex-grow: 1` already on both children, simply hiding `.grid-pane` (`display: none`) lets `.list-pane` naturally expand to fill 100% of the row via ordinary flexbox grow distribution, no manual flex-basis recalculation needed. Toggled via `#splitView.classList.toggle('code-mode', show)` in `_showCodeView()`, with CSS `.split-view.code-mode .grid-pane { display: none; }`. The draggable splitter handle itself lives inside `<node-projects-split-view>`'s *own* shadow root (`[part="splitter"]` in `SplitView.js`), so hiding it from our component requires the CSS Shadow Parts API across the boundary: `.split-view.code-mode::part(splitter) { display: none; }` (works because the split-view element itself, `#splitView`, sits in *our* shadow root, and it exposes `part="splitter"` on its internal handle).
